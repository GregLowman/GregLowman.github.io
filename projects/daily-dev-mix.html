Full-Stack Web App Case Study

Daily Dev Mix

A Spotify companion that builds playlists from your actual listening behavior — not algorithmic guesses.

Node.jsExpressPostgreSQL HandlebarsSpotify APIDocker RailwayMocha + Chai
View on GitHub Live Demo

Overview

Most music recommendation systems tell you what to listen to based on aggregate history or demographic data. Daily Dev Mix takes a different approach — it watches what you actually listen to during a specific activity and builds a playlist from that session's data.

The result is playlists that are genuinely tied to context. The music you listen to while working out is different from what you listen to while studying, even if it's the same artist. Daily Dev Mix captures that distinction by separating listening sessions by user-defined "vibes" and building each playlist independently over time.

My Contribution

This was a six-person team project. My primary responsibilities were:

  • Spotify API integration — designed and implemented the full OAuth 2.0 authorization flow, token management, playback polling, recommendations querying, and playlist creation
  • Railway deployment — configured and deployed both the Node.js app and the PostgreSQL database to Railway, including environment management and the Docker Compose setup
  • Application architecture — established the initial Express structure, route organization, and Handlebars shell that the team built on top of
  • Project management — coordinated team workflow, sprint planning, and milestone delivery

Technical Deep Dive

Application Flow

The app follows a clean MVC pattern. Users authenticate via Spotify OAuth 2.0, create named "vibes" (activities), and start listening sessions. During a session, the app polls /v1/me/player/currently-playing at regular intervals, logging each track to PostgreSQL tied to that session and vibe. When the session ends, artists heard during that session are extracted and passed as seed data to the Spotify Recommendations API, which returns a curated set of tracks that is then written back to the user's Spotify account as a playlist.

Session & Token Management

Spotify access tokens are stored entirely server-side using Express session middleware — kept off the client to prevent exposure. Each protected route validates session token existence before proceeding. All Spotify communication is handled through a centralized service layer, keeping API logic cleanly separated from route handlers.

Database Schema

Four core entities: users, vibes, sessions, and tracks. This structure allows full reconstruction of listening history for any vibe, artist extraction, and seed data generation at session end. The schema is initialized on first Docker run via an SQL init script.

Containerization & Deployment

Docker Compose orchestrates two containers locally — the Node.js app and the PostgreSQL database. The same configuration forms the basis of the Railway deployment, keeping local and production environments consistent.

Takeaways

This project gave me real experience with OAuth flows and third-party API integration at a level most coursework doesn't reach. Managing access tokens securely server-side, handling token expiration gracefully, and coordinating a multi-container deployment taught me things I wouldn't have learned any other way.

If I were building this again, I'd implement token refresh logic more robustly from the start rather than retrofitting it, and I'd invest earlier in a more structured integration test suite for the Spotify layer specifically.
