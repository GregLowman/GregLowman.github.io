Daily Dev Mix

A Spotify companion that builds personalized, activity-based playlists from your real listening behavior — not algorithmic guesses.

Node.jsExpressPostgreSQL HandlebarsSpotify APIOAuth 2.0 DockerRailwayMocha + Chai
View on GitHub Live Demo

Overview

Most music recommendation systems tell you what to listen to based on aggregate listening history or demographic data. Daily Dev Mix takes a different approach — it watches what you actually listen to during a specific activity and builds a playlist from that session's data.

The result is playlists that are genuinely tied to context. The music you listen to while working out is different from what you listen to while studying, even if it's the same artist. Daily Dev Mix captures that distinction by separating listening sessions by user-defined "vibes" and building each playlist independently over time.

Note: The live demo requires a valid Spotify account. The app is currently in Spotify developer mode, which limits access to pre-approved accounts. Open an issue on GitHub or reach out directly to request access.

Technical Deep Dive

Session & Token Management

Spotify access tokens are stored entirely server-side using Express session middleware — never exposed to the client. Each request to a protected route validates that a valid session token exists before proceeding. This approach keeps tokens off the client entirely and ensures authenticated context persists across requests without requiring re-authorization.

Spotify Integration Layer

All Spotify API communication is handled through a centralized service layer, cleanly separated from route handlers. This layer manages the full lifecycle of a session's music data:

  • OAuth authorization redirect and token exchange
  • Polling /v1/me/player/currently-playing at regular intervals during active sessions
  • Extracting unique artists from session track logs at session end
  • Passing artists as seed data to /v1/recommendations
  • Creating a playlist via /v1/users/{id}/playlists and populating it with recommended tracks

Spotify API Endpoints Used

Endpoint Purpose
/authorizeOAuth 2.0 authorization redirect
/api/tokenAccess token exchange
/v1/meFetch authenticated user profile
/v1/me/player/currently-playingPoll playback state during session
/v1/recommendationsGenerate tracks from artist seeds
/v1/users/{id}/playlistsCreate playlist in user's account
/v1/playlists/{id}/tracksAdd recommended tracks to playlist

Database Schema

Four core entities power the application's data model:

users
  • spotify_user_id primary key
  • display_name
  • access_token
vibes
  • vibe_id primary key
  • user_id foreign key
  • name
sessions
  • session_id primary key
  • user_id foreign key
  • vibe_id foreign key
  • start_time, end_time
tracks
  • track_id primary key
  • session_id foreign key
  • spotify_track_id
  • track_name, artist_name

Containerization & Deployment

Docker Compose orchestrates two containers locally — the Node.js app server and the PostgreSQL database. The database initializes on first run via an SQL init script that creates all tables and seeds default data. The same Docker configuration forms the basis of the Railway deployment, keeping local and production environments consistent.

Testing

The test suite uses Mocha as the test runner and Chai for assertions, covering core route behavior and Spotify API integration points. Tests run automatically on docker-compose up and can be run independently with npm test.

Takeaways

This project gave me hands-on experience with OAuth 2.0 at a level most coursework doesn't reach. Managing access tokens securely server-side, handling the full authorization code flow, and coordinating a multi-container deployment taught me things I wouldn't have learned building toy projects. The Spotify integration layer in particular forced me to think carefully about separation of concerns — keeping API logic isolated from route handlers made the codebase significantly easier for the rest of the team to work with.

If I were building this again I'd implement token refresh logic more robustly from the start rather than retrofitting it, and I'd invest earlier in a more structured integration test suite specifically for the Spotify layer. The polling interval logic for playback state also has room for improvement — a websocket approach would be cleaner than repeated HTTP polling.
